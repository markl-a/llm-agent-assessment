### 1. RAG（檢索增強生成）的原理與應用

> **📋 原始題目**
>
> - 請用簡單的方式解釋 RAG 的核心概念，並說明在資訊檢索與內容生成上的優勢。
> - 舉一個您認為最具代表性的 RAG 應用場景，以及該場景中的關鍵挑戰有哪些？
> - 說明幾種不同的 RAG 技術？（Self-RAG, CAG, Graph RAG, KG…）
> - 請簡單闡述 RAG 不能解決的問題為何？
#### (a) RAG 核心概念與雙面優勢

**一句話理解**：RAG（Retrieval-Augmented Generation，檢索增強生成）就是「開卷考試」——先讓 LLM 到外部知識庫「翻書」，取回最相關的段落塞進 prompt，再讓模型「根據找到的資料」作答，而非純靠訓練時壓進參數的記憶（parametric knowledge）憑空回答。

**運作流程（兩階段）**

```text
[離線 / 索引階段]
原始文件 → 切塊(chunking) → 嵌入模型(embedding) → 向量 → 存入向量資料庫(Vector DB)

[線上 / 查詢階段]
使用者提問 → 問題嵌入 → 向量相似度檢索(Top-K)
           → (選配 rerank) → 檢索片段 + 問題組成 prompt → LLM 生成帶引用的答案
```

1. **索引 / 嵌入**：把文件切成片段（可用固定長度，或語意／結構感知切塊），用 embedding 模型轉成高維向量；在良好 embedding 的理想情況下，語意相近的內容在向量空間中距離也相近（這是 embedding 的目標／假設，而非必然成立）。
2. **向量檢索**：把使用者問題轉成向量，以 cosine / dot-product 等相似度做最近鄰檢索（大規模時多用 ANN 近似最近鄰），取回相似度最高的 Top-K 片段——相似度高未必等於最相關，故常再加一層 rerank。
3. **增強生成**：把檢索片段當作 context 連同問題餵給 LLM，指示它「僅根據提供的資料回答並附上出處」，由 LLM 完成語意整合與自然語言生成。

**優勢（分「檢索」與「生成」兩面向）**

| 面向 | 優勢 | 說明 |
|------|------|------|
| 資訊檢索 | 知識時效性 | 更新知識只需重建 / 增量更新索引，不必重訓模型，可快速（近即時）反映最新資料，延遲取決於增量／重建索引的更新頻率 |
| 資訊檢索 | 可控知識邊界與權限 | 檢索範圍可依租戶 / 角色做 ACL 過濾，使提供給模型的 context 僅含授權語料；仍需 prompt 約束與輸出過濾，以降低模型以內部（參數）知識越界回答（parametric leakage）的風險 |
| 資訊檢索 | 成本效益 | 中小型模型 + 檢索，往往比把知識全部灌進超大模型再微調更便宜 |
| 內容生成 | 可溯源／可稽核 | 答案可附帶原文出處，便於人工查核與稽核；但引用未必等於模型實際的推理依據（post-hoc citation），citation 忠實度本身需驗證（citation faithfulness），不宜逕稱「可解釋性」 |
| 內容生成 | 降低幻覺 | 在檢索品質良好且強制依 context 作答時，通常可明顯降低憑空捏造，但不保證歸零，且錯誤檢索反而可能引入新錯誤 |
| 內容生成 | 領域熱插拔、避免遺忘 | 領域知識外掛化、可熱插拔，換領域只換語料即可，不動模型權重，避免 fine-tune 的災難性遺忘與重訓維運負擔（與「成本」不同的獨立面向） |

一言以蔽之：**檢索負責「找到正確且授權的事實」，生成負責「用自然語言把事實講清楚」**；兩者解耦後，可同時兼得時效性、可控性與可溯源／可稽核性。

#### (b) 代表性場景：企業知識庫／法遵合約問答（Enterprise Knowledge Base / Compliance QA）

最具代表性的是**企業知識庫 / 法遵合約問答**。它同時具備高價值（省下大量人工查找）、資料私有（公開模型的參數知識無法覆蓋）、高正確性要求（答錯有法遵或營運風險），最能突顯 RAG 的價值，也最能暴露其工程難點。

**典型語料**：內部規章 SOP、合約、產品文件、Confluence / SharePoint、工單與客服紀錄、法規條文。

**關鍵挑戰（皆可用工程手段優化）**

- **檢索品質與召回率**：檢索品質是 RAG 品質的上限——在嚴格限定只依 context 作答時，檢索失敗會導致無法作答或品質明顯下降。專有名詞、縮寫、同義詞會造成 embedding 無法準確捕捉語意（領域／詞彙不匹配、OOV），需搭配 **hybrid search（BM25 稀疏檢索 + dense 向量）**，以 RRF（Reciprocal Rank Fusion）融合，兼顧關鍵字精確匹配與語意召回。
- **Chunking 策略**：切太大稀釋語意、浪費 context；切太小造成語意斷裂、跨段資訊被切開。實務上採 **結構感知切塊 + 適度 overlap**，保留標題 / 章節 metadata，或用 parent-child（小塊檢索、回填大塊）與 contextual retrieval（每塊前置一段約 50–100 token、將其定位於全文脈絡的情境說明）改善。
- **Embedding 與 Reranker 選型**：需評估中英 / 多語能力、領域適配、向量維度（影響儲存與延遲）。Reranker（cross-encoder，如 bge-reranker / Cohere Rerank）對 Top-K 做二次精排，在多數語料上性價比高，但需權衡其延遲／算力（GPU）成本。
- **多跳推理（multi-hop）**：如「A 合約引用的 B 附件在 C 版本下的責任歸屬」，單輪檢索難以一次抓齊，需 query decomposition、迭代檢索或 Agentic RAG。
- **資料權限、治理與隱私**：必須做 **document-level ACL**，檢索前依使用者身分過濾以避免越權洩漏；ACL 需向量庫支援 metadata / pre-filter，且高選擇性過濾可能傷害 ANN（如 HNSW）召回，須在 pre-/post-filter 間權衡（post-filter 可能把 Top-K 濾到過少甚至全空）。同時處理版本控制（合約以最新生效版為準），避免引用過期條文。此外還有**資料落地／隱私治理**：機密語料送第三方 embedding／LLM API 的外洩風險、PII 去識別、GDPR／個資法合規、機密語料改採地端／私有部署、審視供應商資料使用條款，並注意嵌入向量本身可能被反推還原（embedding inversion）的風險。上線後索引仍需持續增量更新、版本控管與 ACL 維護，是長期營運而非一勞永逸的專案。
- **RAG 特有安全風險**：檢索內容會進入 prompt，需防範**間接提示注入（indirect prompt injection）**——攻擊者把惡意指令藏進被檢索的文件；以及**知識庫資料投毒（data poisoning）**。對策為對檢索片段做輸入淨化與指令隔離、明確區分「資料 vs 指令」邊界、來源信任分級，並對可寫入語料做投毒防護與稽核。
- **評估困難**：難點在需先自建**黃金題組**；有標註後，檢索側用 Recall@K / MRR / nDCG 與 context precision / context recall，生成側用 faithfulness（忠實度）、answer relevancy，常用 RAGAS、LLM-as-a-judge 搭配黃金題組執行。
- **幻覺**：即使有 context，模型仍可能腦補未出現的內容，需強制「僅依 context 作答、無依據就說不知道」並做引用比對驗證。
- **延遲與成本**：檢索 + rerank + 長 context 生成會疊加延遲，長 context 也推高 token 成本，需在召回品質、K 值、模型大小與快取間權衡。

#### (c) 各種 RAG 技術比較

| 技術 | 維度 | 核心機制 | 適用情境 |
|------|------|----------|----------|
| **Naive RAG** | 成熟度演進 | 最基本流程：切塊 → 嵌入 → Top-K 向量檢索 → 直接塞入 prompt 生成 | 快速 PoC、語料乾淨、問題單純的基準線 |
| **Advanced RAG** | 成熟度演進 | 在檢索前後加優化：pre-retrieval 做 query rewriting / 擴展 / hybrid search；post-retrieval 做 reranking、上下文壓縮去雜訊 | 生產級系統的標準做法，需明顯提升召回與精度 |
| **Modular RAG** | 成熟度演進 | 把 RAG 拆成可重組模組（search、rewrite、rerank、memory、routing、fusion 等），以流程編排彈性串接、迭代或條件分支 | Advanced RAG 的架構化演進，需靈活組裝多種策略的複雜系統 |
| **Self-RAG** | 具體技巧（自我反思） | 訓練 LLM 產生 reflection token，讓模型自行決定「是否需檢索、檢索內容是否相關、生成是否有依據」並自評 | 需模型自主控制檢索時機、強調答案自我批判 |
| **CRAG（Corrective RAG）** | 具體技巧（糾錯） | 加一個 retrieval evaluator，將檢索品質分三態：**correct** 先對片段做 decompose-then-recompose 知識精煉（去雜訊）、**incorrect** 才 fallback 網路搜尋、**ambiguous** 兩者並用，再生成 | 檢索品質不穩、需自動糾錯與 fallback 的高可靠場景 |
| **Adaptive RAG** | 具體技巧（路由） | 先用分類器判斷問題複雜度，動態選路：簡單題直接答、中等題單次檢索、複雜題走多輪迭代檢索 | 問題複雜度差異大、要在延遲 / 成本與品質間動態取捨 |
| **RAG-Fusion** | 具體技巧（查詢擴展） | 一問生成多個變體查詢，各自檢索後以 RRF 融合排序 | 使用者問法模糊、單一查詢召回不穩，需以多查詢提升召回 |
| **HyDE** | 具體技巧（查詢擴展） | 先讓 LLM 生成「假想答案」，再用其嵌入去檢索 | 關鍵字與語料措辭落差大、問句過短，需拉近查詢與文件的語意 |
| **Graph RAG** | 資料源／架構 | 用 LLM 從語料抽取實體 / 關係建圖，做社群偵測與階層摘要；檢索時走圖結構 + 摘要 | 需全域彙總、跨文件主題歸納、「整體在講什麼」類問題 |
| **KG-RAG（Knowledge Graph RAG）** | 資料源／架構 | 以既有結構化知識圖譜為檢索源，透過圖查詢（如 Cypher / SPARQL）取回精確關係三元組 | 關係明確、需精確多跳事實推理（如法規、金融、生醫本體） |
| **CAG（Cache-Augmented Generation）** | 架構選擇（以快取取代檢索） | 免即時檢索：將整份知識預先塞入長上下文，離線算好並復用其 KV cache，查詢時省去線上檢索延遲 | 知識庫小到可放進上下文窗口（context window）、低延遲要求、語料相對靜態；代價是每次查詢仍須對整份長 context 做注意力運算（解碼延遲／記憶體開銷）、有 lost-in-the-middle 品質衰減，且 KV cache 與模型／精度綁定，適用前提除「知識庫小」外還需權衡查詢頻率與 context 長度 |
| **Agentic RAG** | 架構選擇（agent 導向） | 以 agent 為核心：具規劃、工具調用、多來源路由與多步迭代，能自主決定檢索什麼、何時停止、是否呼叫外部工具 | 複雜多跳、跨資料源、需結合計算 / API 的開放式任務 |

> 補充：CAG 與 API 層的 **prompt caching** 兩者底層其實是**同一種機制**（皆為前綴 KV cache 復用），真正分野在定位與實作層級：CAG 由使用者**離線算好、長期復用**，用以**取代檢索**；prompt caching 是 API 層對重複提示前綴的**短期**快取（多附 TTL），用以省去重複計算、降低延遲與成本，勿只以「目的不同」一語帶過。

**選型建議**

- **從簡到繁**：先用 Naive RAG 建基準與評估集，再視瓶頸升級到 Advanced RAG（reranker + hybrid search 幾乎是必備）。
- **依問題型態選路**：問「明確事實」→ 向量 / KG-RAG；問「整體歸納 / 主題彙總」→ Graph RAG；問法多變模糊 → RAG-Fusion / HyDE。
- **依關係複雜度**：需精確多跳關係推理選 KG-RAG；需開放式跨源多步推理選 Agentic RAG。
- **依可靠性與延遲**：對錯誤敏感選 CRAG / Self-RAG 做自我糾錯；追求極低延遲且知識庫夠小，用 CAG 免檢索。
- **省成本**：用 Adaptive RAG 讓簡單題不做無謂檢索，把預算留給難題。
- 這些技術**並非互斥**，實務常組合（如 Agentic RAG 內部同時用 hybrid search + reranker + CRAG 糾錯）。

#### (d) RAG 無法解決的問題

先界定範圍：本節談的是**單純向量檢索式（naive / retrieval-only）RAG** 的先天限制。其中一部分可用 (c) 的 Graph / Agentic / KG-RAG 等進階變體**緩解**，另一部分則連進階變體也**無法根治**。核心觀念是：RAG 是「知識外掛」，不是「推理修復器」。

**一、單純檢索式 RAG 力有未逮，但可用進階變體緩解（非死局）**

- **複雜跨文件推理與數值計算弱**：需跨多份文件彙總、統計、精確算術或邏輯演繹的問題，靠 Top-K 檢索片段拼湊容易出錯；這類應交給工具 / 程式碼執行或 Agentic RAG，而非讓 LLM 心算。
- **彙總 / 多跳類問題偏弱**：「整份文件在講什麼」「跨十份報告的趨勢」「A 合約引用的 B 附件在 C 版本下的責任歸屬」這類全域性 / 多跳問題，單輪 Top-K 局部檢索難以覆蓋，需 query decomposition、迭代檢索、Graph RAG / KG-RAG 等特化方案。
- **即時運算 / 需呼叫工具的任務**：即時股價、下單、寄信、跑計算等動作型任務，單純檢索無能為力，需 tool calling / Agentic RAG。
- **Chunk 邊界造成語意斷裂**：固定長度切塊可能把完整論述切開，導致檢索到殘缺片段、上下文遺失；此邊界效應**可緩解（結構感知切塊、overlap、parent-child、contextual retrieval）但無法完全消除**。

**二、連進階變體也無法根治的本質限制**

- **無法修正 LLM 本身的推理錯誤**：就算 context 完全正確，模型仍可能誤讀、曲解或推理跳步；RAG 提供事實，但不保證正確推理。
- **來源本身錯誤或互相矛盾**：RAG 忠實反映語料，若原始資料過期、錯誤或彼此衝突，模型會照單全收甚至無所適從；資料品質治理是前提，而非 RAG 能自動修復的。
- **無法 100% 保證零幻覺**：即使有依據，模型仍可能加入未出現的內容或過度延伸，只能降低而非消除，仍需引用驗證與人工把關。

**小結**：RAG 解決的是「知識從哪來、是否時效、可否溯源」；它解決不了「推理對不對、來源本身可不可信、能否保證零幻覺」。至於多跳、彙總、需工具與即時運算等，naive RAG 力有未逮，但可用 Graph / Agentic 等變體緩解；實務上仍需搭配資料治理、工具調用（Agentic）、驗證機制與人工審核，才能構成可靠系統。

#### (a) 什麼是 Agent、感知—規劃—行動迴圈與 ReAct

**Agent 的定義**

AI Agent 是以 LLM 作為「推理核心（reasoning engine）」的系統：它不只是單次問答，而是能在目標導向的迴圈中自主判斷「下一步該做什麼」，並透過呼叫外部工具與環境互動，逐步逼近目標。與單純的 prompt→completion 不同，Agent 具備狀態、能多輪迭代、能依觀察結果動態調整策略。

**何時才需要 Agent（先分層，再決定）**

不是所有任務都該做成 Agent。由簡到繁分三層，優先選能滿足需求的最簡層級：

- **單次呼叫**：分類、摘要、抽取、問答等一問一答。
- **工作流（workflow）**：多步驟但流程由程式碼掌控，模型只負責各節點。
- **Agent**：任務開放、難以事前完整規格化，需模型自行決定軌跡。

決定是否升級到 Agent 前，檢查四個條件：**複雜度**（多步、難預先規格化）、**價值**（值得更高成本與延遲）、**可行性**（模型勝任）、**錯誤成本**（可被測試 / 審查 / 回滾攔截）。任一為否，就退回較簡單的層級。

**感知—規劃—行動迴圈（Perceive–Plan–Act loop）**

Agent 的核心是一個持續運轉的控制迴圈：

- **感知（Perceive）**：把環境狀態納入 context——使用者輸入、前一步工具回傳的結果、記憶檢索到的內容等。
- **規劃（Plan）**：模型基於當前 context 推理，決定要不要用工具、用哪個、帶什麼參數，或是否已可直接產出最終答案。
- **行動（Act）**：實際執行決策——呼叫工具 / API、寫入記憶、或輸出回覆；執行結果又成為下一輪的感知輸入。

迴圈反覆進行，直到達成終止條件（任務完成、超過步數上限、需要人工介入等）。

**ReAct（Reasoning + Acting）**

ReAct 是實作上述迴圈最經典的樣式，把「推理」與「行動」交錯進行，每一步大致對應三個區塊：

```
Thought:      我需要知道台北今天的天氣，應該呼叫天氣查詢工具
Action:       get_weather(location="Taipei")
Observation:  {"temp": 28, "condition": "多雲"}
Thought:      已取得資訊，可以回答使用者了
Answer:       台北今天多雲，氣溫約 28°C
```

- **Reasoning（Thought）**：模型顯式寫出推理過程，等於「出聲思考」，讓決策更穩定、也更可觀測（可記錄與除錯）。
- **Acting（Action）**：把推理落地成一個結構化的工具呼叫。
- **Observation**：工具執行結果回饋給模型，成為下一輪推理的依據。

ReAct 的價值在於：推理引導行動、行動的真實結果又校正推理，**顯著降低**模型憑空幻想的機會（因為它拿得到外部世界的真實回饋）。但要注意：工具回饋只能大幅降低而非消除幻覺——模型仍可能杜撰或誤讀工具結果，因此每個宣稱都應對照工具結果做 grounding，必要時再加一個獨立的 verifier 把關（呼應 (f)）。現代模型的 **adaptive thinking**（extended thinking 的 budget_tokens 版本已在旗艦模型如 Opus 4.8/4.7、Sonnet 5、Fable 5 淘汰，送出會回 400；思考深度改由 `output_config.effort` 控制）可視為把「Thought」內化為模型的思考區塊，並在工具呼叫之間交錯進行（interleaved thinking）。

**其他規劃樣式與單/多代理取捨**

ReAct 只是眾多 agent 樣式之一，實務上會依任務特性換用不同規劃樣式：

- **Plan-and-Execute**：先讓模型一次規劃出完整步驟，再逐步執行。適合流程較可預測、想省 token 與往返的任務，但對中途變化的適應力較弱。
- **Reflexion**：在每次嘗試後加入自我反省與修正記憶。適合可多次嘗試、有明確成敗訊號的任務。
- **Tree-of-Thoughts**：讓模型展開並評估多條推理分支再擇優、可回溯。適合搜索空間大、需要探索的問題，但成本較高。

架構層面還要在**單一 agent** 與**多代理（orchestrator-worker）**間取捨：單一 agent 簡單、快取友善、除錯容易，適合工具面聚焦的任務；多代理讓 orchestrator 拆解任務並分派給專精的 worker（見 (d) 的路由式架構），適合可平行、領域分明或工具面過大的任務，但要付出協調、通訊與快取失效（換模型/換工具面）的成本。

**與外部工具 / API 的互動流程**

Agent 與工具互動是一條標準管線：

1. **工具描述（tool schema）**：以結構化格式提供工具的 `name`、`description`、`parameters`（JSON Schema）給模型。
2. **模型決策**：模型讀取 schema，決定呼叫哪個工具與各參數的值，輸出一個結構化的呼叫請求。
3. **執行**：由外部程式（harness）真正執行該工具 / API。
4. **回饋**：把執行結果（成功資料或錯誤）以工具結果的形式塞回 context。
5. **迭代**：模型根據結果決定繼續呼叫下一個工具，或收斂成最終答覆。

#### (b) Structured Output（結構化輸出）的原理

**問題背景**

LLM 天生輸出自由格式文字。但要把模型接進真實系統（存資料庫、觸發 API、串接下游流程），需要「保證可解析、符合預期 schema」的輸出。若只靠 prompt 拜託模型「請回傳 JSON」，仍可能出現多餘前後綴、逗號錯誤、欄位缺漏、型別錯誤，導致 `JSON.parse()` 失敗。Structured Output 的目標，是依做法不同把「格式正確」從純機率，提升到（在 constrained decoding／`strict: true` 下）可保證的層級。

**常見做法（由弱到強）**

| 做法 | 原理 | 保證強度 |
|---|---|---|
| Prompt 指示 + few-shot | 在 system/prompt 中描述格式並給範例 | 弱（僅靠模型自律） |
| JSON mode | 開啟「只輸出合法 JSON」模式（各家支援不一） | 中（保證合法 JSON，不保證符合特定 schema） |
| Schema 約束 / constrained decoding | 在解碼階段用（由 JSON Schema 編譯出的、上下文無關的）文法／下推自動機限制每一步只能取樣合法 token | 強（保證結構與型別合法；數值/長度等語意約束需另補） |
| 工具呼叫即結構化輸出 | 把「要抽取的結構」定義成一個工具，模型輸出的工具參數本身就是結構化資料 | 強（且語意清晰） |

**Constrained decoding / grammar 約束（核心機制）**

最本質的做法是在**解碼階段**下手：把目標 JSON Schema 編譯成一個文法（JSON 可任意深度巢狀，屬**上下文無關文法**，需以**下推自動機**識別；有限狀態機無法處理無界巢狀），在模型每一步取樣 token 時，遮罩（mask）掉所有會違反文法的 token，只允許合法延續。這樣可**保證輸出的結構與型別合法**，而不是事後驗證再重試——這也是它遠比「純 prompt 拜託」可靠的原因。但要注意這只約束**結構與型別**：數值範圍、字串長度等**語意約束**（Anthropic structured outputs 明確不支援 `minimum`/`maximum`/`multipleOf`、`minLength`/`maxLength`，也不支援遞迴 schema）仍需在 client 端補驗證；此外若 `stop_reason` 為 `refusal` 或 `max_tokens`，輸出仍可能不符或被截斷。

**工具呼叫即結構化輸出**

Function Calling 本質上就是一種 Structured Output：工具的 `input_schema` 是一份 JSON Schema，模型會依 `input_schema` **盡力產生**符合該 schema 的呼叫參數。要注意：未開 `strict: true` 時，工具呼叫僅**盡力遵循** schema、並非硬性保證；若要「嚴格符合」的硬保證，需在工具定義加 `strict: true`（需搭配 `additionalProperties: false` 與 `required`）。因此「強制模型抽取某個結構」常直接用一個工具（或 strict tool use）來實作。

以 Claude 為例：

- 在請求上指定 `output_config: {format: {...}}` 帶入 JSON Schema（舊的頂層 `output_format` 參數已淘汰）；或在工具定義上設 `strict: true`（需搭配 `additionalProperties: false` 與 `required`），即可保證輸出嚴格符合 schema。
- SDK 建議直接用 `client.messages.parse()`，會自動依 schema 驗證並回傳型別化物件。
- 注意限制：**與 Citations 併用會回 400**；首次使用新 schema 有一次性編譯延遲，之後 24 小時內快取；若 `stop_reason` 為 `refusal` 或 `max_tokens`，輸出可能不符 schema 或被截斷，需另行處理。

**為何對系統整合重要**

- **可靠解析**：下游程式不用寫防禦性 parser 或正則去「猜」模型輸出。
- **契約明確**：schema 就是模組間的介面契約，型別安全。
- **降低錯誤重試成本**：省去「解析失敗→重問模型」的迴圈，減少延遲與 token 花費。
- **易於串接自動化**：結構化資料能直接進資料庫、觸發 API，是 Agent 化的前提。

#### (c) Function Calling（工具呼叫）的原理

**核心觀念：模型決定 WHEN / WHICH，程式負責 HOW**

Function Calling 最關鍵、也最容易被誤解的一點是：**模型本身不會、也不能真正執行任何函式**。模型做的只是「決定要呼叫哪個工具、帶什麼參數」，並把這個意圖以結構化格式輸出；真正的執行由模型以外的程式完成。

- **模型負責**：WHEN（此刻該不該用工具）、WHICH（用哪個工具）、以及參數該填什麼。
- **程式負責**：HOW（真正去打 API、查資料庫、執行運算），以及執行結果如何回饋。

這個分工是安全與可控的基礎——模型只是提出請求，執行方可在執行前做權限檢查、參數驗證、使用者確認。

不過「由誰執行」還要再細分兩類工具，別一概而論：

- **Client-side（客戶端）工具**：如自訂工具、bash、text editor、memory——模型回傳 `tool_use`，由開發者的 **harness** 在自己的環境執行，再把 `tool_result` 塞回 context。以下「運作流程」描述的就是這一類。
- **Server-side（供應商端）工具**：如 Claude 的 web search、code execution——由 **Anthropic 的基礎設施直接執行**，並非開發者 harness；模型與工具結果的往返在伺服器端的取樣迴圈中完成，開發者不需自己接。這類工具有伺服器端迴圈的迭代上限（預設 10 次）：達到上限時回應的 `stop_reason` 會是 **`pause_turn`**，此時把該回應原封帶回、再發一次請求即可讓伺服器**續跑**（不要額外加一句「Continue.」，模型會依尾端的 `server_tool_use` 自動接續）。

**運作流程**

1. **提供工具定義**：把每個工具的 `name`、`description`、`input_schema`（JSON Schema）放進請求。

```json
{
  "name": "get_weather",
  "description": "查詢指定城市的目前天氣。當使用者詢問天氣、氣溫或天候時呼叫。",
  "input_schema": {
    "type": "object",
    "properties": {
      "location": { "type": "string", "description": "城市名稱，例如 Taipei" }
    },
    "required": ["location"]
  }
}
```

2. **模型輸出結構化呼叫請求**：模型判斷需要用工具時，回傳一個 `tool_use` 區塊（而非真的執行），同時回應的 `stop_reason` 標示為 `tool_use`：

```json
{ "type": "tool_use", "id": "toolu_abc123",
  "name": "get_weather", "input": { "location": "Taipei" } }
```

3. **外部程式執行**：harness 讀到這個請求，實際呼叫天氣 API。

4. **把結果餵回模型**：以 `tool_result`（對應同一個 `tool_use_id`）送回：

```json
{ "type": "tool_result", "tool_use_id": "toolu_abc123",
  "content": "28°C，多雲" }
```

5. **模型繼續**：模型讀到結果後，決定收斂成最終回答，或再呼叫下一個工具。若工具失敗，回傳 `is_error: true` 讓模型能改用其他策略或回報錯誤。

**控制何時使用工具（`tool_choice`）**

| 設定 | 行為 |
|---|---|
| `{"type": "auto"}`（預設） | 模型自行決定是否用工具 |
| `{"type": "any"}` | 強制至少呼叫一個工具 |
| `{"type": "tool", "name": "..."}` | 強制呼叫特定工具 |
| `{"type": "none"}` | 禁止使用工具 |

任一設定都可再加 `"disable_parallel_tool_use": true`，強制每次最多只呼叫一個工具。

**設計要點**

- `description` 要「規定何時該呼叫」而非只描述功能（例如「當使用者問到目前價格或近期事件時呼叫」），能顯著提升該叫就叫的觸發率。
- 支援**平行工具呼叫**：一則 assistant 訊息可含多個 `tool_use`，應並行執行後，把**所有** `tool_result` 放進**同一則** user 訊息回傳；拆成多則會讓模型逐漸不再做平行呼叫。
- 每個 `tool_result` 必須帶對應的 `tool_use_id`，且每個 `tool_use` 都要有一個對應結果，否則後續請求會被拒絕。

#### (d) 面對「非常大量工具 / API」的實務策略

當工具數量從個位數成長到數百上千個，會遇到兩個主要問題：**context 爆掉**（把所有 schema 塞進提示會吃光 context window 並拉高成本）以及**選錯工具**（工具太多、描述相近，模型難以精準挑選）。實務對策如下：

**1. 工具檢索（Tool RAG / 語意檢索）**

不要一次把所有工具 schema 都塞進 context，而是把工具描述做成可檢索的索引（向量嵌入或 BM25）。每次請求先依使用者意圖檢索出最相關的工具子集（Top-K），只暴露這幾個給模型。這是**最常用的手段之一**。

- **但它不是免費午餐**，實務上要面對幾個風險：**召回不足／誤檢**（真正需要的工具沒被檢索到，或檢索到一堆相近但不對的工具）、**檢索延遲**（每次請求多一次向量檢索的往返）、**embedding 品質與索引維護成本**（工具描述改了要重建索引），以及一個「**雞生蛋**」問題——你得先檢索、才知道要哪個工具，但檢索本身就依賴對工具描述的良好標註。
- **降低召回風險的做法**：用 **hybrid 檢索**（向量語意檢索 + BM25 關鍵字）再加一層 **re-rank**，比單一檢索器穩定；並保留**退場策略**——當檢索信心不足或召回失敗時，fallback 到一個上層 router agent 判斷領域，或（工具數量還能負擔時）直接退回全量暴露，避免因漏檢而讓整個任務失敗。
- 以 Claude 為例可用內建的 **Tool Search 工具**（工具的 `name` 為 `tool_search_tool_regex` 或 `tool_search_tool_bm25`；於 `tools` 宣告時帶日期的 `type` 為 `tool_search_tool_regex_20251119` / `tool_search_tool_bm25_20251119`），把其他工具標記 `defer_loading: true`，由模型動態搜尋後只載入相關 schema。注意：Tool Search 工具本身不可 defer，且至少要保留一個非延遲工具，否則會回 400。

**2. 階層式 / 路由式 Agent**

用一個上層 router / orchestrator agent 先判斷任務屬於哪個領域（如「財務」「行事曆」「檔案」），再交給只掌握該領域工具的子 agent。每個子 agent 的工具面（tool surface）小而聚焦，選擇準確率大幅提升。

**3. 工具分組與命名空間（namespacing）**

用一致的命名前綴與分組（如 `calendar.create_event`、`crm.lookup_contact`）降低語意衝突與歧義，也讓路由與檢索更容易。命名清晰本身就能減少模型混淆。

**4. 動態載入（progressive disclosure）**

只在「當下這一步」需要時才把工具細節**按需載入** context，但載入後**保留（附加式）**、不要中途卸除或抽換——因為工具是 prompt 最前段的內容，中途移除／重排會使整段快取失效（見下方第 6 點與 (h)）。搭配 Skills 這類「漸進揭露」機制：預設只放一句工具 / 技能的簡述，模型判斷需要時才讀取完整定義並附加進來。

**5. 用 MCP 統一介接**

以 MCP（Model Context Protocol）作為標準協定，把散落各處的工具 / 資源以 client-server 方式統一暴露，一次實作、到處可用。這讓「大量工具」的管理從「每個整合各寫一份膠水」變成「掛上標準化的 MCP server」，並可在 server 層做分組與權限。

**6. 避免 context 爆掉與選錯工具的其他做法**

- **快取穩定的工具清單**：把（去重、排序後的）工具定義放在 prompt 前綴並做 prompt caching，避免每次重算（見 (h)）。
- **保持工具集合穩定**：工具是 prompt 最前段的內容，中途新增 / 移除 / 重排會使整段快取失效——用 Tool Search「附加」schema 而非「抽換」，才能保住快取。
- **Programmatic tool calling**：讓模型寫程式碼在沙盒中依序 / 迴圈呼叫多個工具，中間結果留在程式裡、不進 context，只把最終結果回饋給模型，大幅省 token 與往返延遲。
- **良好的工具描述與 few-shot 範例**：對易混淆的工具給正例，降低誤選率。

#### (e) 如何打造 Agent 的記憶功能

記憶讓 Agent 從「無狀態問答」變成「有連續性、能累積經驗」的系統。應分層設計：

**1. 短期 / 工作記憶（short-term / working memory）**

- **內容**：當前對話視窗、近期工具結果、當下任務狀態——也就是塞在 context window 裡的東西。
- **特性**：容量受 context window 限制、隨對話結束即消失。
- **管理**：對話一長就要壓縮或裁剪（見下方摘要與 context editing）。

**2. 長期記憶（long-term memory）**

- **內容**：跨 session 需持續保存的知識、使用者偏好、過往事件。
- **實作**：**常見做法之一**是**向量資料庫 + 語意檢索**——把記憶片段做 embedding 存入向量庫（Pinecone、Weaviate、pgvector、Chroma 等），每次需要時用語意相似度檢索出相關片段注入 context（本質上就是對「記憶」做 RAG）。**另一種同樣主流、Anthropic 也採用的是檔案／文件式記憶**——讓 Agent 讀寫一個 memory 目錄 / `.md` 檔（對應 Claude 的 Memory 工具與 Managed Agents 的 Memory Stores，見下方第 6 點），簡單、可稽核、可版本化，長跑 Agent 常與向量檢索並用。

**3. 記憶類型（memory taxonomy）**

| 類型 | 內容 | 例子 |
|---|---|---|
| 情節記憶（Episodic） | 具體發生過的事件、對話歷程 | 「上週使用者要求把報表改成 GAAP 格式」 |
| 語意記憶（Semantic） | 抽象化的事實、知識、偏好 | 「使用者偏好用 Python」 |
| 程序記憶（Procedural） | 做事的方法、流程、學到的技巧 | 「部署前一定要先跑測試」 |

實務上會把不同類型分開存放與檢索，甚至用不同的觸發條件與生命週期管理。

**4. 對話摘要與壓縮（summarization / compaction）**

- 當歷程接近 context 上限時，把較早的內容**摘要**成濃縮版本，保留關鍵資訊、釋放空間。
- **Compaction（摘要式）vs Context editing（裁剪式）**：前者把舊內容濃縮成摘要區塊；後者直接清除過時的工具結果 / 思考區塊而不做摘要。長跑 Agent 常兩者並用。以 Claude 為例，server-side compaction 會在接近閾值時自動摘要，回傳的 `response.content` 陣列中會**包含**一個 `compaction` 區塊；下一次請求要 **append 整段 `response.content`**（其中含該 `compaction` 區塊），而非只取文字、或只取單一 block，否則會遺失壓縮狀態。

**5. 記憶的寫入 / 檢索 / 更新 / 遺忘策略**

- **寫入（Write）**：判斷「什麼值得記」。避免記下 repo 或對話已有的內容；一則記憶一個重點。**特別要守住信任邊界以防「記憶投毒（memory poisoning）」**——攻擊者可藉 prompt injection 讓 Agent 把假事實寫進長期記憶，跨 session 汙染後續決策。因此記憶寫入前要標記**來源／信任等級**，做**可信—不可信分離**：來自外部內容（網頁、工具回傳、他人訊息）的記憶預設視為不可信，需經校驗或人工確認才升級為可信；讀取採信時也依信任等級決定權重。
- **檢索（Retrieve）**：任務開始前先查記憶；用語意檢索取回 Top-K 相關片段，注意相關性排序，避免塞入雜訊。
- **更新（Update）**：對已存在的記憶做修正而非重複新增；可用內容雜湊做樂觀鎖（optimistic concurrency）避免覆寫衝突。
- **遺忘（Forget）**：主動刪除已證實錯誤或過時的記憶；可用 TTL、重要度衰減、或明確的 redact（用於洩漏的機密 / PII）。

**6. 可用框架 / 做法**

- LangChain / LangGraph 的 memory 模組、LlamaIndex 的向量索引、Letta（前 MemGPT）的分層記憶架構。
- 供應商原生機制：Claude 的 **Memory 工具**（讓模型讀寫 `/memories` 目錄，後端儲存由你實作）、Managed Agents 的 **Memory Stores**（跨 session 持久化，附帶版本與稽核，並支援 redact）。
- **安全提醒**：記憶內不要存 API key / 密碼 / token；PII 需遵循 GDPR / CCPA；多使用者系統要做 per-user 記憶隔離與存取控制。

#### (f) 設計 LLM Agent 時需特別考慮的問題

- **可靠性與錯誤處理**：工具會失敗、API 會逾時。要把錯誤（含 `is_error`）結構化回饋給模型，讓它能重試或改道；對 429 / 5xx 做指數退避重試；區分可重試（限流、伺服器錯、網路）與不可重試（400 / 404）錯誤，分別處理。
- **迴圈 / 終止條件與失控防護**：Agent 迴圈必須有明確終止條件——最大步數 / 最大工具呼叫數、預算上限、達成目標、或偵測到重複的無效動作。務必設 `max_iterations` 之類的硬上限，避免無限迴圈燒錢；可用 task budget（讓模型知道剩餘預算並自我節制）。
- **成本與延遲**：多輪 Agent 每步都是一次 LLM 呼叫，token 與延遲會累積。對策：prompt caching、選對模型（簡單子任務用便宜模型 / subagent）、調整 effort 等級、平行化工具呼叫、programmatic tool calling 減少往返。
- **幻覺與正確性**：模型可能杜撰工具結果或進度。用 grounding（要求每個宣稱都對照工具結果）、結構化輸出約束、以及外部驗證步驟（獨立的 verifier subagent）來把關。
- **安全**：
  - **Prompt injection**：外部內容（網頁、檔案、工具回傳）可能夾帶惡意指令。用可信 / 不可信內容分離、operator 專用的 system channel（不可被使用者內容偽造）。
  - **權限最小化**：工具只給任務所需的最小權限；金鑰範圍越小，出事時的爆炸半徑越小。憑證不要進沙盒、不要寫進 prompt 或訊息歷程。
  - **工具濫用**：對高風險、不可逆的動作（刪資料、送訊息、金流）做確認閘門或人工核可；bash / 檔案工具要沙盒化、白名單、路徑校驗。
- **可觀測性與追蹤（observability / tracing）**：記錄每一步的 thought、工具呼叫、參數、結果、token 用量與 request id。ReAct 的顯式推理天然有助除錯；串接 tracing（如 OpenTelemetry / LangSmith）以便回放與稽核。
- **評估（evaluation）**：Agent 難以用單一指標衡量。要建立任務層級的評測集、成功率、成本 / 步數效率。要區分**軌跡評估（trajectory）**（過程中每一步的工具選擇、參數、推理是否合理）與**結果評估（outcome）**（最終產物是否達標）——只看結果會漏掉「僥倖答對」與「過程危險」的情況。實務上採 **離線 golden / eval set + 線上監控** 並行：離線集擋回歸、線上監控抓真實分布的漂移。要注意 Agent 有**非決定性**（同一輸入多次執行軌跡不同），使回歸難以精確比對，需用容忍度 / 多次取樣，或聚焦於結果與關鍵不變量。若用 **LLM-judge**，要正視其本身的偏差（位置偏好、冗長偏好、自我偏好等）並做**校準**——用人工標註對齊、固定 rubric、隨機化選項順序、必要時多 judge 投票。
- **版本控管與回歸測試（prompt / 工具 / 模型）**：換模型、改 prompt、增減工具都會改變 Agent 行為（也會使 prompt cache 失效）。要對 prompt、工具定義、模型版本做**版本鎖定**與變更紀錄，任何一項變動都跑一次回歸評測（對照上面的 golden set）；在非決定性下，回歸要用統計性判準（多次取樣的成功率、關鍵不變量）而非逐字比對，避免「改了 prompt 或換了模型，線上行為才悄悄變了」這類實務盲點。
- **狀態管理**：API 本身無狀態，每次都要送完整歷程。要妥善管理對話狀態、記憶、以及跨 session 的持久化與版本。
- **人類介入（human-in-the-loop）**：對高風險決策插入確認關卡（approval gate），或在 Agent 卡住 / 不確定時升級給人。用 manual loop 而非全自動 tool runner 以取得逐步核可能力。
- **冪等與副作用控制**：工具重試可能重複執行副作用（重複下單、重複送信）。設計冪等（idempotency key）、避免同一動作被重放；區分唯讀（可平行、可重試）與有副作用（需序列化、需去重）的工具。

#### (g)（加分）MCP 與 Function Calling 的關係

**兩者位於不同層級，是互補而非取代**

| 面向 | Function Calling | MCP（Model Context Protocol） |
|---|---|---|
| 層級 | **模型層能力** | **應用 / 整合層的開放標準協定** |
| 本質 | 模型決定「呼叫哪個工具、帶什麼參數」 | 標準化「如何把 tools / resources / prompts 暴露給應用」 |
| 提出者 | 各 LLM 供應商的通用能力 | Anthropic 提出的開放標準 |
| 架構 | 請求中的 tool schema + `tool_use` | client-server 架構 |
| 類比 | 模型「會用工具」的這項能力 | AI 工具界的「USB-C」——一次實作，到處可用 |

**Function Calling 是模型層能力**

Function Calling 描述的是「模型有能力依 schema 決定並輸出一個結構化工具呼叫」。它關心的是模型與工具的**介面**（schema、`tool_use`、`tool_result`）這一層。

**MCP 是開放標準協定**

MCP 用 client-server 架構，把工具（tools）、資源（resources）、提示範本（prompts）以**標準化**方式暴露給任何支援 MCP 的應用（host / client）。它解決的是「整合」問題：以前每接一個外部系統就要為每個 LLM 應用各寫一份膠水程式；有了 MCP，只要實作一個 MCP server，任何 MCP client 都能用——「一次實作，到處可用」，就像 USB-C 讓各種裝置以同一介面互通。

**兩者如何協同**

關鍵在於：**MCP server 暴露的工具（tools），最終仍以 function-calling 的 schema 形式呈現給模型**。流程是——MCP client 從 server 取得工具清單 → 轉換成模型看得懂的 tool schema → 模型用 Function Calling 決定要呼叫哪個 → client 透過 MCP 協定把呼叫轉發給對應 server 執行 → 結果沿原路回饋給模型。

但別忘了 MCP 暴露的是**三類**能力，而且**只有 tools 走 function-calling 的決策路徑**；另外兩類以不同機制進入請求，不經模型的「要不要呼叫工具」決策：

- **Resources（資源）**：作為**上下文（context）注入**——由 client / host 讀取後，以文件／內容區塊的形式放進請求 context 供模型參考（等同餵資料，而不是讓模型決定去「呼叫」它）。
- **Prompts（提示範本）**：作為**範本**——由使用者或應用主動選用後，展開成預設的訊息序列注入請求（是一種可重用的 prompt 模板，同樣不經 function-calling 決策）。

所以更精確地說：MCP 負責「工具 / 資源 / 範本從哪裡來、怎麼標準化地接進來」，其中 tools 依賴 Function Calling 作為模型端的選用機制，resources 與 prompts 則透過 context 注入 / 範本展開進入請求。MCP 不取代 Function Calling，反而依賴它作為模型端的決策機制；兩者堆疊在一起，才構成完整的「標準化整合 + 模型自主選用」能力。

#### (h)（加分）Prompt Caching 如何幫助 Agent，以及其原理

**原理：快取 prompt 前綴的 KV cache**

Transformer 處理輸入時，會為每個 token 算出注意力機制所需的 Key / Value 向量（即 KV cache）。Prompt Caching 的核心是把 prompt **前綴**的這些中間運算快取起來；當後續請求的前綴與之相同時，就能**省去重算前綴的 KV**，直接載入快取。要注意這不代表「完全不碰前綴」——新增尾段的每個 token 在生成時**仍需 attend 前綴的 KV**，只是那些前綴 KV 是從快取讀出、不必重新計算而已。

因此它是一種**前綴比對（prefix match）**機制：快取鍵由「渲染後 prompt 的實際位元組」到某個 breakpoint 為止決定。渲染順序是 `tools → system → messages`——只要前綴中任何一個位元組改變，該位置之後的所有快取全部失效。

**適合快取的內容**

- system prompt（穩定的角色設定與指令）
- 工具定義（大量工具的 schema）
- RAG 灌入的大段脈絡 / 檢索文件
- few-shot 範例

這些都是「跨請求重複、且量大」的內容，正是快取的最大受益者。

**效益：大幅降低延遲與成本**

- **快取讀取（cache read）**：約僅需原始 input 價格的 **0.1×**（一成）。
- **快取寫入（cache write）**：5 分鐘 TTL 約 **1.25×**、1 小時 TTL 約 **2×** 原價。
- **損益平衡**：以 5 分鐘 TTL 為例，同一前綴被用到第 2 次就開始划算（1.25× + 0.1× = 1.35× < 2× 未快取）；1 小時 TTL 因寫入成本翻倍，通常需 3 次以上才回本。

**TTL 與門檻（以 Claude 為例）**

- **TTL**：預設 5 分鐘（`{"type": "ephemeral"}`），或 1 小時（`{"type": "ephemeral", "ttl": "1h"}`）。TTL 內只要有命中就會續期。
- **最小可快取前綴（依模型而異）**：Opus 4.8 / 4.7 / 4.6 / 4.5、Haiku 4.5 為 **4096** tokens；Fable 5、Sonnet 4.6、Haiku 3.5 / 3 為 **2048**；Sonnet 4.5 / 4.1 / 4 / 3.7 為 **1024**。低於門檻即使加了標記也不會快取（無錯誤，`cache_creation_input_tokens` 為 0）。
- **breakpoint 上限**：每個請求最多 **4** 個 `cache_control` breakpoint。

```python
# 把穩定不變的大段 system prompt 放前面並標記快取
response = client.messages.create(
    model="claude-opus-4-8",
    max_tokens=1024,
    system=[{
        "type": "text",
        "text": LARGE_STABLE_SYSTEM_PROMPT,     # 工具定義 / RAG 脈絡 / few-shot 也放這段前綴
        "cache_control": {"type": "ephemeral"},  # 預設 5 分鐘 TTL
    }],
    messages=[{"role": "user", "content": user_question}],  # 變動內容放最後
)
# 用 usage 驗證命中：
print(response.usage.cache_read_input_tokens)      # 命中快取的 token（約 0.1x）
print(response.usage.cache_creation_input_tokens)  # 寫入快取的 token（約 1.25x）
```

**為何 Agent 場景特別受惠**

- **多輪迭代**：Agent 每一步都要重送整段歷程（API 無狀態）。歷程前綴穩定、命中率高，省下的成本與延遲隨步數線性累積。
- **長 system prompt + 大量工具定義**：這些是每一步都重複的固定前綴，快取後每步都省——尤其是 (d) 提到的「大量工具」場景，工具 schema 一旦穩定就是理想的快取內容。
- **RAG / 長脈絡**：檢索到的大段文件若在多次追問間共用，快取一次、後續多次讀取，效益顯著。

**設計原則：穩定不變的內容放前面**

由於是前綴比對，設計上要把「絕不改變」的內容（凍結的 system prompt、去重 / 排序後的工具清單）放最前面、跨越 breakpoint；把「每次都變」的內容（時間戳、UUID、per-request ID、當前提問）放到最後一個 breakpoint 之後。

常見的「沉默失效」陷阱：在 system prompt 裡插入 `datetime.now()`、未排序的 `json.dumps()`、隨使用者變動的工具集——這些都會讓前綴每次都不同，快取永遠 0 命中。可用 `cache_read_input_tokens` 是否為 0 來偵測是否有沉默失效。

**幾個進階但關鍵的細節**

- **失效有層級**：快取分 tools / system / messages 三層，變更只影響「自己這層及以下」。因此改 `tool_choice`、切換 `thinking`、換圖片等不會動到 tools+system 快取；但**增刪 / 重排工具或更換模型會強制整段重建**（快取綁定模型）。
- **20 個區塊回溯窗**：每個 breakpoint 最多往回找 20 個 content block 尋找先前快取。單一回合若加入超過 20 個區塊（agentic 迴圈常見的大量 tool_use / tool_result），下一次可能靜默 miss——在長回合中每約 15 個區塊補一個中間 breakpoint。
- **fork 要沿用父請求前綴**：摘要、compaction、subagent 等分岔運算，要**原封不動**沿用父請求的 `system` / `tools` / `model`，才能命中父請求的快取。
- **可預熱（pre-warm）**：對延遲敏感且前綴龐大的互動場景，可在啟動時送一個 `max_tokens: 0` 的請求先寫入快取，消除第一個真實請求的冷啟延遲。注意 `max_tokens: 0` **不可與下列組合併用**（否則回 `invalid_request_error`）：`stream: true`、`thinking.type: "enabled"`（啟用 thinking）、`output_config.format`（結構化輸出）、強制 `tool_choice`（`{"type":"tool"}` 或 `{"type":"any"}`）、以及 Message Batches；`cache_control` 要標在與真實請求共用的前綴（system / 工具定義）最後一個 block，而非標在 placeholder 訊息上。

### 3. Prompt Engineering 的重要性

> **📋 原始題目**
>
> 在使用生成式 AI 時，Prompt Engineering 極為關鍵。請描述您對 Prompt Engineering 的理解，以及哪些要素（例如：上下文、角色、輸出格式、指令明確度）最能影響模型輸出品質。

#### Prompt Engineering 的本質

Prompt Engineering 是在「模型權重固定」的前提下，透過設計輸入（指令、上下文、格式約束與範例），把模型從龐大的可能輸出分布中，引導到你真正想要的那一小塊高品質區域。它的本質不是「找魔法咒語」，而是**降低任務歧義、對齊模型的解讀與你的意圖，並約束輸出的形狀**。

從機率語言模型的角度理解會更清楚：LLM 做的是條件生成 `P(output | prompt)`；更精確地說，它是逐 token 的自回歸生成 `∏ P(token_t | prompt, token_<t)`，最終輸出還會受解碼取樣參數（temperature、top-p 等）影響，而非由 prompt 單獨決定。Prompt 的每個元素都在改變這個條件分布的形狀——好的 prompt 讓高品質答案的機率質量集中、讓錯誤路徑的機率下降。因此**多數與「內容設計」相關的技巧可歸納為三件事**：**提供足夠證據（context）、限縮解空間（格式與約束）、示範目標分布（範例與推理過程）**。要留意這個三分法只涵蓋「內容設計」層面；另有成本／延遲優化（如 Prompt Caching）與工程化技巧（迭代式評估、版本管理）並不落在這三者任一項之內，後文會分別討論。

#### 最能影響輸出品質的四大要素

以下四要素是本題指定的分析框架，可與上述「本質三支柱」對照著看：**上下文**對應支柱一（提供證據）；**輸出格式**與**指令明確度**共同對應支柱二（限縮解空間）；**角色設定**則是獨立於三支柱之外的「校準層」，負責把輸出分布往特定語域偏移；而支柱三（示範目標分布，即範例與推理）因屬進階手段，留待後文〈進階技巧〉討論。換句話說，範例既非「不重要」、也非核心四要素之一，而是可按需疊加的進階技巧——這樣兩套框架就能對接，不致混淆。

**1. 上下文（提供足夠且相關的背景／資料）**

- **為什麼影響品質**：模型的參數化知識有截止日期，且對「你的特定情境」一無所知。缺乏上下文時，模型只能依賴記憶去「猜」，這是幻覺（hallucination）與答非所問的**重要成因之一**。把相關事實放進上下文，等於把任務從「開放式記憶題」變成「有據可查的閱讀理解題」，正確率大幅提升。但要注意：補足上下文並不能根除幻覺——即使提供了上下文，模型仍可能違背上下文而捏造（faithfulness hallucination，不忠於來源的幻覺），RAG 場景照樣會發生，因此仍需搭配「只依提供內容作答」的指示與輸出校驗。
- **關鍵在「相關」而非「多」**：一方面，無關資訊會稀釋注意力、增加雜訊，反而降低品質；另一方面，長上下文還存在一種與「相關資訊擺放位置」有關的位置性偏差——lost-in-the-middle：相關資訊被放在上下文中段時較容易被忽略（這是位置本身造成的，與周圍是不是無關資訊無關）。兩者結合的實務建議是：上下文應精選、去重，並把最關鍵的資料放在開頭或結尾，而非中段。

| 面向 | 壞範例 | 好範例 |
|---|---|---|
| 內容 | 「幫我判斷這筆交易有沒有問題？」 | 「以下是我們的退款政策（附全文）與這筆交易明細（附 JSON）。依政策判斷此交易是否符合退款資格，並指出對應條款編號。」 |
| 問題 | 模型不知道「政策」是什麼，只能編 | 有明確依據，可追溯、可驗證 |

**2. 角色設定（persona／system prompt）**

- **為什麼影響品質**：角色設定會把輸出分布往特定語域、專業深度與價值取向偏移。指定「資深資安工程師」與不指定，模型對同一問題的用詞精確度、考慮的邊界情況、風險意識都不同。system prompt 還承載**全域且穩定的規則**（語氣、禁忌、輸出語言）。在部分經過指令階層（instruction hierarchy）訓練的模型中，system 通常比一般 user 訊息優先，適合放偏好性的全域約束；但這**並非硬保證**——該特性主要來自特定廠商的訓練（如 OpenAI 於 2024 年提出的 instruction hierarchy），並非所有模型都具備，且 system prompt 常被 jailbreak／prompt injection 繞過。因此**安全關鍵的約束不能只靠 system prompt**，必須再配合輸出過濾、外層 guardrail 等縱深防禦。
- **注意**：角色設定是「校準器」不是「事實來源」。給模型戴上「醫師」帽子不會讓它擁有它沒有的知識，真正的專業知識仍需靠上下文或檢索補足。過度戲劇化的 persona（「你是全世界最強的天才」）通常沒有實質幫助，具體的專業框架（「依 OWASP Top 10 逐項檢查」）才有效。

```text
壞：你是一個 AI 助理，請回答問題。
好：你是一位熟悉 PCI-DSS 的資安稽核員。審視程式碼時，
   針對每個發現點標註嚴重度（高/中/低）、對應條款與修補建議。
   只根據提供的程式碼作答，無法判斷時明確說「資訊不足」。
```

**3. 輸出格式（明確指定結構、JSON、範本、長度）**

- **為什麼影響品質**：其一是**可用性**——下游若要程式解析，自由文字幾乎不可用，明確的 JSON／表格 schema 讓輸出可被機器消費。其二是**引導思考**——指定欄位等於替模型列出「必須考慮的面向」，例如要求輸出 `{風險, 依據條款, 建議}` 會促使模型針對這三個面向各給一項回應。但要清楚：「產出欄位」不等於「實質推理」——模型也可能只為填滿格式而給出淺薄或杜撰的值，因此宜搭配思考空間／CoT（見下）以降低為填格式而杜撰的風險。其三是**控制長度**能避免冗長或過度省略。
- **實務要點**：JSON 場景應提供明確 schema、要求「只輸出 JSON、不要加說明文字或 markdown 圍欄」；能力允許時，優先使用平台原生機制而非純靠 prompt 求 JSON，可靠度顯著提升。**一個已知的反面 trade-off**：過於嚴格的格式約束在需要深度推理的任務上可能壓抑推理品質（模型把生成資源花在滿足語法、而非思考）；補救方式是讓「推理欄位」排在最終結構化欄位之前，或分兩段執行——先自由推理，再把結果格式化。這裡有三個常被混用的名詞需要辨析：

| 名詞 | 定義 | 目的 |
|---|---|---|
| Function Calling／Tool Use | 模型輸出「呼叫哪個工具、帶哪些參數」的結構化請求，由外部程式執行後把結果回填 | 讓模型驅動外部行動 |
| Structured Output（結構化輸出） | 約束模型的最終回覆嚴格符合指定的結構（最常見為 JSON Schema，亦可為正規式、CFG/EBNF 文法、枚舉等） | 讓輸出可被程式穩定解析 |
| Constrained Decoding | 在解碼階段限制可取樣的 token，強制輸出符合指定語法／schema | 是 Structured Output 嚴格符合 schema 的核心機制；對 function calling 則是提升參數可靠度的可選（strict）增強 |

> function calling 的參數本身也可視為 schema 約束的結構化輸出，兩者目的不同：前者是為了「觸發外部行動」，後者是為了「產出可解析的答案」。要辨析的是：function calling 能運作的**根本能力來自模型微調**（模型學會輸出工具呼叫），constrained decoding 只是保證參數嚴格符合 schema 的可選（strict）增強，而**非** function calling 成立的基礎；反過來，對 Structured Output 而言，constrained decoding 才是「嚴格合規」的核心機制。

| 壞範例 | 好範例 |
|---|---|
| 「分析一下這些評論的情緒。」 | 「針對每則評論輸出一個 JSON 物件，欄位：`id`(int)、`sentiment`(positive/neutral/negative)、`confidence`(0–1)。整體以 JSON 陣列回傳，不要輸出其他文字。」 |

**4. 指令明確度（具體、可執行、避免模糊）**

- **為什麼影響品質**：模糊指令會讓模型在解讀階段就發散。「盡快」「簡短」「專業一點」都是主觀詞，模型只能猜你的標準。把模糊詞換成可量化、可執行的約束，等於直接縮小解空間。
- **明確度的具體手段**：量化（「200 字以內」而非「簡短」）、給判準（「若信心低於 0.7 則標記為需人工複核」）、界定範圍（「只依提供的文件作答」「只輸出繁體中文」）、指定步驟。其中有三點須加註意：
  - **長度約束是軟性引導**：模型並不精確計數，中文字數尤其不準，「200 字以內」通常只被大致遵守。應容忍偏差；若有硬性上限，需在後處理階段以程式校驗與裁切。
  - **模型自陳信心未必校準**：直接拿模型自報的 confidence 當 0.7 門檻並不可靠——LLM 的自陳信心通常校準很差，模型說「0.85」並不對應 85% 的真實正確率。門檻應以實測資料校準，或改用更可靠的信號，如 token logprob、自一致性（self-consistency）投票等。
  - **範圍界定要挑模型真能遵守的條件**：像「只依提供的文件作答」「只輸出繁體中文」模型可實際遵守；反之「只考慮 2024 年後的資料」對純參數化知識並不可靠——LLM 對其內部知識沒有可靠的時間戳，無法真正過濾內部記憶。這類時間過濾只有放到 RAG／檢索層（依文件日期過濾）才成立。

```text
壞：幫我把這段文字改得好一點。
好：把以下產品描述改寫為適合電商列表的版本。要求：
   1) 繁體中文；2) 80–120 字；3) 開頭一句賣點、結尾一句行動呼籲；
   4) 不得杜撰規格，只能使用原文提供的資訊。
```

#### 進階技巧

**Few-shot 範例（示範目標分布）**
在 prompt 中放數組「輸入→期望輸出」，**在不改動模型權重的前提下，經驗上是教格式與風格最有效的手段之一**，尤其當任務難以用文字規則窮舉時（例如特定標註慣例）。這一點也因任務而異：若是大量且穩定的格式需求，微調往往比 few-shot 更有效、也更省 token。要點：範例要有代表性、涵蓋邊界情況（含反例）、格式與正式請求完全一致；範例間標籤分布要平衡，否則會引入偏誤（如多數標籤偏向）。若零樣本已足夠，就不必付出 token 成本。

**Chain-of-Thought／逐步推理**
對多步推理、數學、邏輯類任務，要求模型「先逐步推理再給結論」能顯著提升正確率，因為它把運算攤在生成的 token 上，而非要求一步得出答案。但這有代價：CoT 會增加輸出 token、拉高延遲與成本，對簡單任務可能無益、甚至有害（不必要的推理反而引入錯誤），因此不宜對所有任務一律強加 CoT。實務上：

- 需要乾淨輸出時，讓推理走在指定的思考區（如 `<thinking>…</thinking>`）或 scratchpad，最後再輸出結論欄位。
- 新一代具備原生推理能力的模型，往往不需要、甚至不宜再手動塞入僵化的 CoT 指令，過度指定反而干擾其內部推理；應以任務效果實測為準。

**任務拆解（decomposition）**
複雜任務拆成數個單一職責的子步驟（或多次呼叫串接），每步的解空間更小、更容易正確，也更好偵錯與快取。例如「擷取 → 分類 → 彙整 → 產出報告」分階段執行，對複雜任務通常優於一個巨型 prompt 一次做完。但多次呼叫也有代價：累加的延遲與成本、跨階段的誤差傳播（前一段錯，後段可能全錯）、以及編排與狀態管理的複雜度上升——其中尤以誤差傳播需特別留意。這也是 prompt chaining 與 agent 設計的基礎。

**分隔符／標籤界定內容**
用 XML 標籤、Markdown 圍欄或明確分隔線把「指令」與「資料」分開，能降低模型把使用者資料誤當指令執行的機率，並讓模型精準引用某段內容。但要清楚：分隔符對 prompt injection **只是弱防護**，不是可靠防線——攻擊者可以注入偽造的結束標籤、或塞入「忽略上述指令」之類的越界內容繞過它，純分隔符對刻意注入幾乎沒有實質防護力。真正的防護需搭配輸入淨化、權限最小化、輸出後檢查等縱深防禦。

```text
<policy>…政策全文…</policy>
<transaction>…交易 JSON…</transaction>
指令：僅依 <policy> 判斷 <transaction> 是否符合退款資格。
```

**給模型思考空間**
不要在還沒推理時就逼它下結論。安排「先分析、再結論」的順序，或明確允許「資訊不足時可回答無法判斷」，能減少為了填滿格式而杜撰。

**負面約束（不要做什麼）**
明確列出禁止事項可堵住常見失敗模式：「不要杜撰未提供的數據」「不要輸出程式碼以外的說明」「引用不到來源時不要回答」。要注意兩點：負面約束越具體越有效，模糊的「不要出錯」無意義；且盡量**同時給出正向替代**（「不要用被動語態，改用主動語態」），純否定有時較難遵循。

**結構化 prompt 以善用 Prompt Caching**
Prompt Caching 會快取一段**前綴**的內部運算結果，後續請求只要前綴「逐字相同」即可命中，降低延遲與輸入端成本。它是基礎設施層的優化，但實務上有幾個常被忽略的細節與成本條件：

- **有最小可快取前綴長度門檻**：前綴太短（依模型約 1K–4K token 不等）不會被快取，且**不會報錯**，只是靜默失效。
- **快取會過期（TTL）**：預設約 5 分鐘，可選 1 小時；過期就得重寫。
- **多數平台需顯式設定 cache breakpoint**（`cache_control` 斷點），而非自動全快取。
- **寫入快取有溢價、僅讀取才便宜**：寫入約為基礎輸入價的 1.25x（1 小時 TTL 約 2x），讀取約 0.1x。因此**淨收益取決於重用頻率**——必須在 TTL 內被多次讀取、重用頻率足夠高才划算；一次性請求反而更貴。

能否奏效也取決於 prompt 的結構安排：

- 把**穩定內容**（系統指令、工具定義、few-shot 範例、大型參考文件）放在前段，作為可快取前綴。
- 把**每次變動的使用者輸入**放在最後。
- 這與 lost-in-the-middle 的擺放建議大致不衝突：龐大且穩定的參考資料置於開頭（可快取），真正的問題置於結尾（近因位置、利於作答）。但要補一句：整塊參考資料置於開頭，並不代表它「自身的中段」就免疫——大型參考資料**內部**仍有中段片段被忽略的風險，因此其中的關鍵片段宜在該段落內再往前置或後置、或加上標註、重排以突顯，而非只靠整體位置。

**迭代式評估與 prompt 版本管理**
Prompt Engineering 是實證工程，不是一次寫成。應建立：

- **評估集（eval set）**：一組代表性輸入＋期望輸出或評分準則，可用精確比對、規則檢查或 LLM-as-judge 做自動化評分。採用 LLM-as-judge 時需警覺其已知偏誤——位置偏誤、對冗長回答的偏好、對自家模型的自我偏好等，應設法控制（如打亂順序、隱藏來源），並定期以人工抽檢校準評審本身。
- **可重現的比較**：盡量固定 `temperature`（甚至設為 0）等取樣參數後 A/B 比較不同 prompt 版本。`seed` 也常被當成可重現性旋鈕，但它的可得性因平台而異——Anthropic Messages API 並沒有 `seed` 參數，OpenAI 的 `seed` 也僅為 best-effort、官方不保證完全可重現；因此不宜把 seed 當作跨平台的可重現性保證。此外即使固定了這些參數，分散式推論仍可能殘留少量非決定性。這麼做是為了避免用單一漂亮案例自我說服。
- **版本控制**：把 prompt 當程式碼管理（納入 git、標註版本號、記錄變更理由與對應指標），並將模型版本一併記錄——換模型可能讓舊 prompt 失效（prompt 對模型有耦合性）。
- **回歸測試**：改 prompt 修好 A 案例時，跑全套 eval 確認沒弄壞 B、C 案例。

#### Prompt Engineering 的限制與風險

前面談了 prompt 能做什麼，這裡誠實補上它做不到、以及容易踩雷的地方——這也正是「為什麼永遠要靠 eval 守品質」的原因：

- **對措辭與範例順序極度敏感**：換一個近義詞、調動 few-shot 範例的先後，輸出就可能明顯改變。prompt 的 robustness（穩健性）通常不好，看似無關的小改動也可能讓結果退化。
- **報酬遞減**：一段 prompt 調到某個程度後，繼續加指令、加約束的邊際效益快速下降，甚至互相打架（過多「CRITICAL／MUST」反而讓模型過度觸發）。
- **沒有形式化保證**：prompt 是引導而非約束，無法從理論上保證模型一定遵守；再好的 prompt 也只是把出錯機率壓低，不是壓到零。
- **跨模型／版本脆弱**：prompt 對特定模型有耦合性，換模型或模型升級都可能讓原本可用的 prompt 失效，必須重新評估。
- **結論**：正因為以上，prompt 工程本質是實證工程——必須用前述的 eval set 與回歸測試把品質「守」住，不能靠一次寫成或主觀感覺自我說服。

#### 與 RAG、Fine-tuning、Agent 的關係與界線

四者不是互斥的替代品，而是**不同層次的槓桿**，解決不同性質的落差。實務上常組合使用，選擇原則可由「缺的是什麼」來判斷：

| 面向 | 解決的核心問題 | 成本／延遲 | 何時採用 |
|---|---|---|---|
| Prompt Engineering | 意圖對齊、格式、行為引導 | 最低、即時 | 永遠先做；問題是「模型能力夠、但沒被正確引導」 |
| RAG（檢索增強） | 知識缺口、時效性、可追溯來源 | 中；加檢索延遲與基建 | 需要私有／即時／大量且會變動的知識，且要求引用來源 |
| Fine-tuning | 穩定的風格／格式／領域行為、把長 prompt 內化 | 一次性訓練／資料／維運成本高；但推論延遲／成本可能反而更低（微調常能縮短 prompt） | 行為需求穩定且量大，靠 prompt 難以穩定達成，或要降低推論成本 |
| Agent 設計 | 多步驟、需工具與外部行動、動態決策 | 最高；編排、狀態、可靠性挑戰 | 任務需規劃、呼叫工具、與環境互動、跨步驟決策 |

**判斷準則**

- **先窮盡 prompt**：多數「模型表現不好」其實是引導不足。改 prompt 成本最低、迭代最快，應是第一手段，也是評估其他方案的基準線。
- **知識不足 → RAG**：症狀是「答錯事實、內容過時、需要引用來源、涉及私有資料」。這是**知識問題不是能力問題**；微調來灌事實既昂貴又易過時，正解是檢索。RAG 把正確資料放進上下文，因此**本身極度依賴 prompt**——如何組織檢索結果、如何指示模型「只根據檢索內容作答」，都是 prompt 工程。常見變體可依需求選用：
  - **Naive RAG**：檢索 → 拼接 → 生成，最基本形態。
  - **Advanced RAG**：加入檢索前後處理，如查詢改寫（query rewriting）、重排序（re-ranking）、壓縮與去重，改善召回與精度。
  - **Agentic RAG**：由 agent 動態決定是否檢索、檢索幾次、如何組合多個來源（含多輪與多工具檢索）。
  - **GraphRAG**：以知識圖譜組織實體與關係，擅長需要跨文件彙整、全域摘要的問題。
- **行為不穩 → Fine-tuning**：症狀是「特定格式／語氣／領域慣例反覆教不會，或 prompt 已臃腫到難維護、成本過高」。這是**行為與風格的問題**。微調能把示範內化進權重，換取更短的 prompt 與更穩定的輸出，但無法解決時效性知識缺口（那仍是 RAG 的工作）。
- **需要行動與多步流程 → Agent**：症狀是「單次生成無法完成，需要呼叫工具、查資料、規劃、依中間結果決定下一步」。Agent 是**編排層**，其每個節點（規劃、工具選擇、反思）內部仍是一次次 prompt 呼叫，因此**扎實的 prompt 工程與任務拆解是 agent 可靠性的地基**。在工具接入層，**MCP（Model Context Protocol）** 提供一個開放標準，用統一的 client–server 介面規範「工具與上下文如何被描述、發現與存取」；它與 function calling 分工不同——function calling 是模型「決定呼叫哪個工具、帶什麼參數」的機制，MCP 則標準化這些工具與資料源如何被提供給模型，兩者互補而非取代。

一句話總結界線：**Prompt 調「怎麼問與怎麼答」、RAG 補「答案所需的知識」、Fine-tuning 改「模型內建的行為傾向」、Agent 決定「如何分步行動與使用工具」**。四者是疊加的能力層——先用 prompt 把能力榨到極限，再依「缺知識、缺穩定行為、還是缺行動能力」分別引入 RAG、fine-tuning 與 agent；而且無論走到哪一層，prompt 工程都始終是底層共用的基本功。

### 4. 聊天機器人應用情境（電商商品客服）

> **📋 原始題目**
>
> 假設客戶是一間電商公司，需要利用一個 LLM 模型與客戶進行商品客服的對話。請問：
>
> 1. 你會怎麼實作與設計 Agent？
> 2. 請舉出 Agent 對電商公司的價值為何？為什麼 Agent 特別適合電商公司？
> 3. 你打算如何處理「模型幻覺（Hallucination）」問題，使模型回答更精準？
> 4. 如何評估客服聊天機器人的效能、使用體驗，以及最終對業務帶來的成效？

#### (1) Agent 架構設計

整體採「**Orchestrator + 專職元件**」的分層架構：由對話總控（Orchestrator / Router）決定每一輪要做意圖判斷、走 RAG、呼叫工具、還是轉人工。關鍵是 orchestrator 是一個 **agentic loop**——它可以在**同一輪內混合並多次呼叫 RAG／工具**，並把每次結果**回饋到決策節點**再決定下一步，藉此處理「查訂單＋查退貨政策」這類混合意圖與多跳工具呼叫。各元件單一職責、可獨立測試與替換。

**各層設計重點**

| 層 | 設計方式 |
|---|---|
| 意圖判斷/路由 | 把**意圖分類**與**情緒偵測**拆成兩個正交維度並行判斷（intent + sentiment/priority），由兩者共同決定路由與 handoff——例如「對訂單延遲的抱怨」同時具備「訂單物流」意圖與「負面情緒」，不會被互斥錯分。意圖枚舉如 `FAQ查詢 / 商品諮詢 / 訂單物流 / 退換貨 / 閒聊`。分類器選型分兩類：**延遲敏感的線上路由用 embedding 分類器或小模型**（低延遲、低成本）；few-shot LLM 只作為困難意圖的後備或離線標註，不放在每輪必經的路由熱路徑上。低信心意圖走 fallback 詢問澄清，可直接 handoff |
| RAG | **靜態知識入向量庫**：商品的規格／材質／使用說明、FAQ、退換貨與運費政策等相對穩定的描述切 chunk 後做 embedding；檢索採 **hybrid search（BM25 + 向量）+ rerank**。**多輪對話先做 query 改寫/濃縮**，將帶指代的追問改寫為獨立查詢再檢索。政策類文件加 metadata（品類、地區、生效日）以過濾。可用 **Anthropic Contextual Retrieval**（chunk 嵌入前補上文脈，含 **Contextual Embeddings 與 Contextual BM25**），效果是**降低檢索失敗率**（搭配 rerank 降幅更大）、提升檢索準確度。檢索結果作為 grounding 上下文，並要求模型標注引用來源。**索引時效治理**：政策／型錄一有變更即時重建索引、做來源新鮮度檢查——並區分「有 grounding」與「grounding 正確」兩層，避免檢索到過期來源時「忠實引用錯誤」 |
| 工具 (Function Calling) | **事實走工具、知識走 RAG** 是核心分界。價格、庫存、上下架、訂單、物流、退款金額等**會變動的動態欄位一律工具即時取得**，例：`get_order(order_id)`、`get_inventory(sku)`、`track_shipment(tracking_no)`、`create_return(order_id, items, reason)`。工具以 **JSON Schema（Anthropic `input_schema`／OpenAI `parameters`）嚴格定義輸入參數**；模型只輸出結構化的呼叫請求，**回傳格式由應用端自行約定並可另做結構驗證**（模型端不校驗工具回傳）。**安全授權**：對所有涉及個資的**讀取型**工具（`get_order`、`track_shipment`）同樣強制登入／OTP 綁定與資料範圍授權（只回傳本人訂單），避免枚舉單號越權查他人訂單／PII（IDOR）；**寫入型**工具（建退貨單）額外需二次確認 |
| 對話記憶 | **短期**：當前對話 window + 結構化 slot（order_id、SKU、語言、已驗證身分）；**長期**：會員檔案、歷史訂單摘要、既往客訴。長對話用 summary memory 壓縮，避免 context 爆量 |
| Guardrails | 輸入端：Prompt injection 偵測、PII 遮罩、越權請求攔截；**間接提示注入防護**：對經 RAG 取回的內容（UGC 評論／Q&A）與工具回傳做**淨化／隔離，標記為「不可信資料」而非指令**（對應 OWASP LLM01），敏感動作輸出前再校驗；輸出端：內容安全、禁止承諾未授權的折扣/退款、價格與政策必須來自工具或 RAG 來源、格式/結構驗證 |
| Handoff | 觸發條件：**連續低信心（初值可設連續 2 次低於信心閾值，再依 CSAT／FCR 回饋調參）**、負面情緒/投訴升級、涉及金流爭議、使用者主動要求、超過 N 輪未解決。**技術實作**：轉接時把**對話摘要、已驗證資訊、建議動作**經 **CRM／工單系統 API**（填入標準欄位）帶入專員工作台，讓真人在既有介面無縫接手 |
| 後端整合 | 工具層封裝為 API Gateway，串接 OMS（訂單）、WMS/庫存、物流商 API、CRM、金流。亦可透過 **MCP（Model Context Protocol）**：讓**應用（host／client）以統一介面連接後端工具與資料源，再供模型使用**（模型不直接連 MCP server）。MCP 的取捨是——在**多系統／多模型**情境能攤提整合成本，但需評估**協定成熟度**與**工具授權、資料外洩面**等安全。設計冪等鍵、逾時/重試、rate limit；全鏈路 tracing（每輪的意圖、檢索、工具呼叫、token）進 observability 平台 |

**資料治理／隱私（PII 遮罩 vs 身分驗證）**

前端「PII 遮罩」與後續需 email／電話／地址做身分驗證與後端查詢並不矛盾——要區分兩條路徑：**送 LLM／記錄層**的資料做去識別遮罩（不讓明碼進 prompt 與 log）；**經安全通道傳後端**的則是原始個資，並以 tokenization／可控還原機制在需要時還原對應，兩者用不同的處理與存取權限。

**強一致資料的快取取捨**

庫存等**強一致資料不快取，或僅快取「近似可用量」供展示**，**下單一律以後端鎖庫存為準**；避免在短 TTL 快取窗口內已售罄卻仍回「有貨」而超賣。要明確標示：哪些資料（靜態型錄描述、政策文本）可快取，哪些（價格、庫存、訂單狀態）須即時取得。

**跨層機制**

- **Structured Output vs Function Calling**：兩者職責不同——**工具參數的約束屬 Function Calling**（模型決定呼叫哪個工具、帶什麼輸入）；**Structured Output 指對一般回覆施加 schema 約束**（如路由決策、slot 抽取輸出成固定 JSON）。「保證可解析」須在 **strict／受限解碼（constrained decoding）** 模式下才近乎成立，一般模式仍應做解析容錯。
- **Prompt Caching**：對穩定的 system prompt、政策文件、工具定義與 few-shot 範例啟用前綴快取。它**只降低「被快取的重複前綴」的 input token 成本並縮短首字延遲，生成（output）token 成本不變**；且有前提——需達**最小可快取 token 長度**、有**寫入加價**（首次寫入約 1.25×，1h TTL 約 2×）、受 **TTL（預設約 5 分鐘，可選 1 小時）過期**與**命中率**影響，故**高流量客服才划算**。快取為**前綴比對、以 token／區塊前綴命中**，前段任何變動都會使其後失效，故將易變內容（時間戳、使用者訊息）置於最後。
- **Streaming**：串流回傳生成內容，縮短使用者「感知延遲」。
- **語意快取**：**僅用於與政策無關的穩定問句**（如「運費怎麼算的一般說明」）。對政策類答案（如退貨流程）不可直接快取「生成後答案」，否則政策更新後仍回舊答案、繞過全篇避免引用過期政策的機制；若要快取，須**綁定政策版本／生效日，並在政策變更時主動失效（cache invalidation）**。
- **延遲/成本預算**：每輪可能串接多次模型／檢索呼叫（意圖、改寫、rerank、生成、輸入／輸出 Guardrail），需給預算與取捨——意圖＋情緒可**合併成一次分類呼叫**、檢索與部分工具可**並行**；意圖分類、Guardrail 等**以 embedding／小模型或規則替代**大模型；用**串流**降感知延遲。設定端到端 **P95 目標**（例如純 FAQ ≤ 2s、需工具查詢 ≤ 4s）作為監控與最佳化依據。

實作上以事件化的 orchestration（如 state machine / graph）串接，讓「判斷 → 檢索/呼叫 → 回饋決策 → 生成 → 驗證」每一步都可觀測、可回放、可加測試，並支援上述一輪多跳與工具結果回饋。

#### (2) Agent 對電商的價值與適配性

**核心價值**

- **24/7 即時服務**：夜間與促銷尖峰不斷線，回應從「數分鐘」壓到「數秒」。
- **規模化與降本**：一個 Agent 同時服務數千對話，**相對真人客服邊際成本大幅下降、可近線性擴充**；將大量重複性問題（訂單在哪、怎麼退貨、有沒有貨）自助化，讓真人客服專注高價值/高情緒案件。要注意主要**變動成本是 token 與後端（檢索／工具）呼叫**——並非零成本，因此才需要搭配 **Prompt Caching** 與**模型分級**（簡單任務走小模型）來控管單位成本。
- **提升轉換與客單價**：不只被動答問，還能主動導購、比較商品、交叉銷售/加購推薦（bundle、湊免運），把客服對話變成營收管道，拉高轉換率與 AOV。
- **多語言一致性**：一套邏輯支援多語系市場，回答口徑（政策、話術、品牌調性）一致，不因客服個人差異而波動。**前提**：主要語系一致性佳，**低資源語言的理解／生成品質仍有落差，需額外在地化與品質驗證／評測**。
- **即時資料整合**：電商價值高度依賴「即時真相」——庫存、訂單狀態、物流節點、會員等級。Agent 透過工具直接查真實系統回答，這是靜態 FAQ 或人工翻系統做不到的效率。
- **資料回饋做洞察**：對話結構化沉澱後，可分析熱門問題、缺貨/退貨主因、話術成效，回饋給商品、行銷與供應鏈決策。

**為什麼「特別」適合電商**：電商同時具備「高流量、高重複性、可標準化政策、強即時系統整合、與營收直接掛鉤」五個特性——正好落在 Agent 最擅長的甜蜜點：可用 RAG 覆蓋政策/FAQ、可用工具接系統拿即時真相、又能在對話中推動成交，投入產出比高於多數行業。

**限制與風險／不適用情境**

Agent 並非全能，以下情境需要保守設計或人工介入：

- **複雜爭議與情緒升級**：多方糾紛、重大客訴，交由真人處理較妥。
- **金流與法遵**：退款、發票、消保等涉及金錢與法律責任的動作，須有審批與範圍界定。
- **幻覺誤導的責任**：若模型錯誤承諾退款或給錯法律/保固資訊，可能造成實質損失——這類高風險輸出要**限制在 handoff 之後由真人確認**。
- **冷啟動資料不足**：新品類或新市場的 FAQ／政策語料不足時，檢索與答案品質不穩。

因應方式：設**信心閾值**，低於閾值即 **handoff**；明確界定 Agent 的**責任範圍與可承諾邊界**（例如金額、政策僅能引用、不能自行裁量），把不確定與高風險案件導向真人。

#### (3) 幻覺（Hallucination）控制策略

原則：**能查就不猜，能約束就不放任，查不到就誠實轉人工。**

- **事實走工具、知識走 RAG（兩條路徑）**：這是防幻覺最關鍵的分界。
  - **知識型（政策細則、FAQ、商品靜態描述）→ RAG grounding**：一律以檢索到的**最新版政策/FAQ/商品文件**為依據，未檢索到支撐內容就不得自行生成事實；prompt 明確要求「僅根據提供的來源作答」。政策細則屬知識，走 RAG（並靠索引時效治理確保檢索到的是現行版本）。
  - **事實型（價格、庫存、訂單狀態、物流、退款金額）→ 呼叫真實系統取得**：這些「會變動且不能錯」的資料一律走工具，禁止模型憑記憶或推測作答。
- **引用與 grounding 檢核**：要求模型輸出所依據的來源片段/文件 ID，方便驗證與稽核；必要時做 grounding check（逐句比對答案是否有來源支撐）。同時區分「有 grounding」與「grounding 正確」——來源本身過期/錯誤時仍可能「忠實引用錯誤」，故需搭配 (1) 的索引時效治理與來源新鮮度檢查。
- **查不到就說不知道 + 轉人工**：明確的 fallback 契約——工具無結果或檢索信心低時，回覆「目前查不到，為您轉接專人」，而非硬編一個答案。
- **輸出後驗證**：生成後做二次校驗，例如比對答案中的數字/SKU/政策是否與工具回傳一致、格式是否合法，不一致則重問或降級轉人工。
- **限縮回答範圍**：系統提示界定 scope（只回答本商城相關問題），越界問題禮貌拒答，降低自由發揮空間（閒聊邊界的處理見 (1) 與下方對齊說明）。
- **降低取樣隨機性（作用有限，非事實依據）**：調低 temperature／收斂 decoding 的作用是**降低同一輸入的答案漂移／不一致**，讓行為更穩定；但它**不能取代 RAG grounding 與工具查證**——「有信心的錯誤」只會被更穩定地重複，並不提供事實依據。部分新一代模型已不提供 temperature 參數，改以結構化輸出、提示與工具約束控制發散；創意場景（導購話術）可適度放寬。
- **Structured Output 約束**：關鍵回覆用 schema / 工具參數約束（如退貨單欄位、可選政策枚舉），讓模型「填空」而非「自由寫作」，從源頭壓縮亂答的可能。

搭配「明確 system prompt 界定角色與禁區 + few-shot 示範拒答樣式」，形成多重防線。

**閒聊邊界與 scope 對齊**：意圖含「閒聊」與「越界（非本商城相關）禮貌拒答」看似衝突，須明定分界——**閒聊（打招呼、輕度寒暄）給簡短友善回應後導回購物**；**與本商城無關的越界問題則禮貌拒答**。兩者是不同處理，與 scope 限縮策略對齊，避免圖文/策略不一致。

#### (4) 客服機器人效能評估（三層指標）

**三層指標框架**

| 層次 | 指標 | 說明 |
|---|---|---|
| 技術/品質 | 回答正確率、幻覺率、**檢索命中/召回（Recall@k / Hit Rate@k）**、**檢索排序品質（MRR / nDCG）**、延遲（P50/P95、首字延遲）、工具呼叫成功率與**工具選擇/軌跡正確率**、Guardrail **攔截率與誤攔率（false positive rate）** | 衡量系統「答得對不對、查得準不準、排序好不好、快不快、安不安全」。命中/召回與排序品質是**不同面向**：Recall@k/Hit Rate@k 看「該找的有沒有被找到」，MRR/nDCG 看「找到的排序好不好」 |
| 使用體驗/營運 | CSAT、首次解決率 FCR、對話放棄率、平均對話輪數、情緒/滿意度趨勢、**containment（未轉人工率）與轉人工率**、**self-service resolution（真正解決率）** | 衡量顧客「用得順不順、問題有沒有一次解掉」。**containment（自助處理率／未轉人工率）與轉人工率互補（containment ≈ 1 − 轉人工率），歸同一層並看**；且 containment 不等於問題真的解決——**self-service resolution（真正解決率）** 須結合 FCR／CSAT 或「後續是否再詢問同一問題」來認定 |
| 業務成效 | 客服成本節省、對話轉換率、AOV、留存/回購率、NPS、每對話貢獻營收 | 衡量對「營收與成本」的真實貢獻 |

> 術語說明：containment 採「自助處理率／未轉人工率」等通行譯法（不用「遏止率」，避免阻止/壓制的誤解），或直接保留英文 containment 並附一致定義。
>
> 注意 containment 與轉人工率要與 CSAT／FCR 並看：一味壓低轉人工可能犧牲滿意度，需平衡而非單點最佳化。

**量測方法**

- **線下（offline）評估**：以真實對話取樣建立黃金測試集並標注，**先做去識別與資料治理（合規流程）** 再使用；每次改 prompt/檢索/模型都跑迴歸測試，比對正確率、幻覺率、檢索命中/召回與排序品質，避免退步。
- **線上（online）評估**：生產環境即時監控 containment、延遲、轉人工率、CSAT，設告警與 dashboard；以 canary/逐步放量控管風險。
- **A/B 測試**：對 prompt、檢索策略、模型版本、話術做分流實驗，以業務指標（轉換率、AOV、成本、CSAT）判定勝出版本，而非只看離線分數；需先估**最小樣本量**並做**顯著性檢定**，避免用不足樣本下結論。
- **LLM-as-a-judge**：用另一個模型依 rubric 自動評分正確性、是否 grounded、語氣與是否誠實拒答。**相對人工標註成本低，適合抽樣或分層覆蓋**（若用「較強（更貴）模型」逐通評分每日全量對話，絕對成本並不低，可改以較小的 judge 模型或抽樣控成本）；需先用人工標註校準 judge 的一致性，並定期抽查防止 judge 偏誤。
- **人工標註**：對敏感、爭議、低信心與抽樣對話由人工評分，作為 ground truth，用來校準 LLM judge、發現盲點，並回饋成新的測試案例與訓練/few-shot 素材，形成「上線 → 監控 → 標註 → 改進 → 再評估」的持續迭代閉環。
